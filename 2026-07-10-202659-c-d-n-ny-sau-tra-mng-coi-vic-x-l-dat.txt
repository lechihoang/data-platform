⏺ Đã đọc xong toàn bộ pipeline và đối chiếu vớiuẩn medallion
  (Databricks/Microsoft). Tổng kết: kiến trúc của bạn về cơ bản là chuẩn, thậm
  chí làm tốt vài điểm nâng cao. Nhưng có một số chỗ lệch giữa code và tài liệu,
  cộng vài vấn đề kỹ thuật cần sửa.

  Đối chiếu với chuẩn medallion

  | Lớp | Chuẩn (Databricks/MS) | Code của bạn | Đánh giá |
  |---|---|---|
  | Bronze | Raw, bất biến, không transform, single source of truth | NiFi đổ
  parquet/csv thô vào s3a://lakehouse/bronze/, không đụng vào | ✅ Đúng chuẩn |
  | Silver | Ép kiểu, làm sạch, dedup, chuẩn hoá schema, validate; không
  aggregate | bronze_to_silver.py: standardize schema → cast → clean → dedup →
  enrich temporal. DQ check bằng Great Expectations | ✅ Đúng chuẩn |
  | Gold | Aggregate theo business, dimensional model, tối ưu query |
  silver_to_gold.py: 4 bảng ag (daily/monthly/zone/payment) | ✅ Đúng chuẩn |

  Điểm bạn làm tốt hơn mức cơ bản:
  - WAP pattern (Write-Audit-Publish) qua Nessie: mỗi run tạo branch riêng → DQ
  gate → MERGE BRANCH vào main. Đây đúng là best practice để không publish data
  bẩn.
  - DQ gate đặt sau cả silver và gold, đúng như MS khuyến nghị ("final data
  quality rules applied in gold").
  - Schema enforcement chặt (STANDARD_TYPES, fill null cho cột thiếu) — đúng
  tinh thần "protect against schema changes".

  Những chỗ cần sửa

  1. Doc và code lệch schema nghiêm trọng (nợ kỹ thuật, dễ gây hiểu lầm)
  docs/nyc_tlc_pipeline_plan.md mô tả gold khác hẳn code thực tế:
  - Doc daily_trips: avg_trip_distance, total_passenger_count, pickup_date
  - Code daily_trips (dòng 33-41): total_distance, total_revenue,
  avg_trip_duration_seconds, trip_date

  monthly_summary, revenue_by_zone cũng lệch tương tự. Nên cập nhật lại doc cho
  khớp code (commit d0a6567 đã sửa notebook nhưng chưa sửa doc).

  2. Thiếu .cache() ở gold (silver_to_gold.py:26)
  df được lọc từ silver.trips rồi dùng lại cho 4 aggregation + 1 count. Không
  cache → Spark quét lại source5 lần. Nên df.cache() sau filter (dòng 30).

  3. .orderBy() vô nghĩa khi ghi bảng (silver_to_gold.py:83)
  revenue_by_zone sort desc(total_revenue) ngay trước khi ghi vào Iceberg
  partitioned table. Thứ tự không được bảo toàn khi lưu mà lại tốn một full
  shuffle. Nên bỏ dòng .orderBy(...) — để việc sort cho tầng query/BI.

  4. Gap nhỏ ở cleansing silver (so với chính doc của bạn)
  - Doc nói lọc outlier total_amount > $10.000, nhưng code chỉ chặn số âm, không
  cap trần (dòng 102-103). Trần chỉ được checkở DQ (max 99999).
  - "Date infiltration" (năm 2001/2088): code không lọc trực tiếp mà dựa gián
  tiếp vào filter Year==target_year & Month==target_month (dòng 128-129). Hoạt
  động được nhưng không tường minh như doc mô tả.

  5. Dedup toàn dòng (bronze_to_silver.py:109)
  dropDuplicates() không key — đúng chuẩn khyến nghị là dedup theo business key.
  TLC không có natural key rõ nên chấp nhận được, nhưng đắt. Cân nhắc dedup
  theo [vendor_id, pickup_datetime, dolocation_id, ...] nếu muốn nhẹ hơn.

  Kết luận: luồng xử lý đúng chuẩn medallion, không có lỗi kiến trúc. Việc cần
  làm là (a) đồng bộ lại doc với code, (b) 2 fix hiệu năng nhanh ở gold (cache +
  bỏ orderBy). Bạn muốn mình sửa luôn phần code gold và cập nhật doc không?
